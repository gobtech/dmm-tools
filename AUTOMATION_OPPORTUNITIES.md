# DMM Tools — Automation Opportunities

## Context

DMM (Dorado Music Marketing) is a LATAM-focused music marketing company managing 50+ artists across 10+ countries with a team of ~15 people. The core workflow for each artist campaign follows a repeatable cycle: pitch DSPs, service radio, chase press coverage, coordinate interviews, run influencer campaigns, and report results back to the client.

Three reporting workflows have already been automated in `dmm-tools`:
1. **Radio Report** — Soundcharts airplay data → formatted .docx
2. **Press Pickup** — 7-source pipeline (RSS feeds + sitemaps + Google News RSS + SearXNG + Serper + Tavily + DuckDuckGo News) → AI relevance filter → social handle classification → DB matching → grouped .docx
3. **DSP Pickup** — 6-platform playlist checker → proof images → .docx

This document identifies the next wave of automation opportunities based on analysis of:
- A sample client proposal (PNAU — LATAM 2026)
- A sample client report (PNAU — weekly update)
- The Master Task Tracker from Notion (221 tasks, ~50 artists)
- An internal highlights email (new contacts/outlets discovery)
- The company website and existing codebase

## Opportunity 1: Client Report Compiler ✅ DONE

### Priority: HIGHEST
### Time saved: ~2-3 hrs per artist per report cycle
### Status: Implemented — `report-compiler/compile_report.py` + "Full Report" tab in web UI

### The Problem

The PNAU Report is a composite document sent to the client (label/management) containing:
1. Release timeline and focus tracks
2. Summary of overall efforts ("Serviced PR", "Pitched to DSPs", etc.)
3. Radio plays by country/station (= Radio Report output)
4. Streaming/playlist highlights with proof screenshots (= DSP Pickup output)
5. Press pickup by country with outlet descriptions and URLs (= Press Pickup output)

Right now, someone runs the three existing tools separately, then manually copy-pastes their outputs into a Google Doc, adds screenshots, writes the intro/efforts section, and formats it as a client-facing PDF. This happens for every artist, every reporting cycle.

### The Solution

A fourth tool — **"Report Compiler"** — that:
- Takes an artist name as input
- Runs Radio Report, DSP Pickup, and Press Pickup in sequence (or pulls from cached results)
- Prompts the user for an optional "efforts summary" (free text) and release timeline
- Composes everything into a single, branded .docx matching the exact format of existing client reports

### Implementation Path

1. **New endpoint**: `/api/report/compile` in `web/app.py`
2. **New module**: `report-compiler/compile_report.py`
3. **Orchestration**: Call the existing `run_press_pickup()`, `run_dsp_pickup()`, and Soundcharts fetch functions programmatically, collecting their structured results (not just text output)
4. **Document assembly**: Use `python-docx` to build a single .docx with:
   - Title page with artist name, campaign period, DMM branding
   - Release timeline section (user-provided or pulled from Google Sheets)
   - "Overall Efforts" section (user-provided bullet points)
   - Radio Plays section — grouped by country/station, play counts
   - Streaming/Playlist Highlights — platform badges, playlist names, follower counts, proof images embedded inline
   - Press Pick Up — by country, outlet name + description + URL (already generated by press tool)
5. **Web UI**: New tab or sub-mode. User enters artist name, selects date range, optionally fills in efforts and timeline, clicks "Generate Full Report"
6. **Proof images**: Embed DSP proof PNGs directly into the .docx (python-docx supports inline images)

### Data Sources
- Soundcharts API (radio plays) — already integrated
- Spotify/Deezer/Apple/Amazon/Claro/YouTube (playlist checks) — already integrated
- Google News/SearXNG/Serper (press) — already integrated
- Google Sheets release schedule — already integrated
- User input for efforts summary and any editorial additions

### Technical Notes
- The three existing tools return structured data (dicts/lists). The compiler consumes these directly rather than re-parsing text output.
- Could add a "quick mode" that only includes sections where data was found (skip empty radio/press sections).
- Consider caching: if the user already ran DSP Pickup today, reuse those results instead of re-scraping.

## Opportunity 2: Proposal Generator ✅ DONE

### Priority: HIGH
### Time saved: ~1-2 hrs per proposal
### Status: Implemented — `proposal-generator/generate_proposal.py` + "Proposal" tab in web UI

### The Problem

Every new artist campaign starts with a proposal document. The PNAU proposal is 8 pages with these sections:
1. Campaign Duration, Focus, Goal/Strategy
2. Timeline (release dates)
3. Digital Marketing (geo-targeted ads, creator campaigns, content strategy)
4. Retail/DSP (pitching strategy per platform)
5. Press targets by country (MX, BR, CL, AR, CO, EC, PE, UY, VE, CR)
6. Radio targets by country
7. DJ and Club Servicing
8. Budget breakdown

The user said **"they all look almost the same."** The structure is identical across proposals — only the artist name, genre context, release dates, collaborator names, and target outlet selections change. Someone spends 1-2 hours copying a previous proposal, find-replacing the artist name, and tweaking the target lists.

### The Solution

A **Proposal Generator** that takes:
- Artist name
- Genre/scene (e.g., "electronic/dance", "indie rock", "pop")
- Release timeline (dates + track names)
- Key collaborators or hooks (e.g., "ft. The Warning", "TV sync with JUMEX")
- Campaign duration
- Monthly budget tier ($2K standard, custom extras)

And generates a complete proposal .docx with all sections pre-filled, including country-specific press and radio target lists filtered by genre relevance.

### Implementation Path

1. **New module**: `proposal-generator/generate_proposal.py`
2. **Template system**: Store the proposal skeleton as a structured template (not a Word template — build programmatically with `python-docx` for full control)
3. **Target databases** (new data files):
   - `data/press_targets.csv` — extend existing press DB with genre tags and tier (key outlet vs. secondary)
   - `data/radio_targets.csv` — new database of radio stations by country, genre affinity, and pricing (from the proposal's radio section)
   - `data/dsp_strategy_templates.json` — per-platform pitching strategies (Spotify, Apple, Amazon, Claro, Deezer, YouTube) with genre-specific playlist targets
4. **Genre-aware filtering**: Given "electronic/dance", pull the electronic-focused press outlets (DJ Mag, Mixmag, Electronic Groove, Beat Night MX, etc.) and radio stations (Beat, Delta 90.3, Energia 97). Given "indie rock", pull different targets.
5. **LLM assist** (optional): Use Claude to generate the Goal/Strategy paragraph and Digital Marketing section based on artist + genre + timeline + hooks. This is the one section that varies meaningfully between proposals.
6. **Budget calculator**: Dropdown selections (which radio stations, influencer tier, club markets) auto-sum the budget section using a pricing lookup table.
7. **Web UI**: New tab or page. Form with artist info → preview → download .docx

### Data Sources
- Press database (existing, needs genre tags added)
- Radio targets database (new — sourced from existing proposals)
- DSP strategy templates (new — extracted from proposal patterns)
- Budget pricing table (new — from proposal budget sections)
- Optional: Claude API for strategy text generation

### Technical Notes
- The press DB already has 1,582 outlets. Adding a genre/category column and a "tier" column (key target vs. supplementary) would enable smart filtering.
- Radio pricing changes infrequently. A simple CSV with station, country, price, duration, genre_affinity would suffice.
- Start simple: v1 could be a fixed template with find-replace. v2 adds genre filtering. v3 adds LLM-generated strategy text.

## Opportunity 3: Weekly Digest / Update Email Generator ✅ DONE

### Priority: HIGH
### Time saved: ~30-60 min per artist per week
### Status: Implemented — `digest-generator/generate_digest.py` + "Weekly Digest" tab in web UI

### The Problem

The task tracker is full of recurring tasks: "Enviar update", "Enviar recap", "Enviar reporte", "Follow up email". For each active artist, someone writes a weekly or bi-weekly email to the label/management summarizing what happened:
- New radio plays this week
- New playlist placements
- Press coverage found
- Upcoming actions / next steps

This is essentially a lighter version of the full Client Report — an email-sized summary rather than a multi-page document.

### The Solution

A **Weekly Digest** generator that:
- Takes an artist name and date range (default: last 7 days)
- Pulls radio plays, playlist placements, and press hits from that window
- Formats a clean email-ready summary (HTML and plain text)
- Includes a "copy to clipboard" button for pasting into Gmail/Outlook

### Implementation Path

1. **New endpoint**: `/api/digest/generate`
2. **Reuse existing tools**: Call the same radio/DSP/press functions with a 7-day window
3. **Output format**: Instead of .docx, generate formatted HTML (for email) and plain text
4. **Template**:
   ```
   Hi [contact],

   Here's this week's update for [Artist]:

   📻 Radio
   - [Station] ([Country]): [Song] — [X] plays

   🎧 Playlist Placements
   - [Playlist] ([Platform], [Followers]) — Position #[N]

   📰 Press
   - [Outlet] ([Country]): [Article Title]
     [URL]

   📋 Next Steps
   - [user-provided bullets]

   Best,
   [Team member name]
   ```
5. **Web UI**: Small form — artist name, period, optional "next steps" text area → generates email preview → copy button
6. **Batch mode**: "Generate digests for all active artists this week" — produces a zip of text files or a single page with all digests

### Technical Notes
- Much faster than the full report since it's just text (no .docx, no proof images)
- Could add a "highlights only" mode that skips sections with no results
- The email template should match DMM's existing communication style (casual professional, bilingual ES/EN)

## Opportunity 4: DSP Pitch Email Drafter

### Priority: MEDIUM-HIGH
### Time saved: ~20-30 min per release per platform

### The Problem

For every release, the team manually writes pitch emails to DSP contacts at Spotify, Apple Music, Deezer, Amazon, Claro Música, YouTube, etc. These emails follow a predictable pattern:
- Intro (artist name, label)
- Release info (track name, release date, genre, Spotify URI, ISRC)
- Why it matters (collaborators, sync placements, radio traction, streaming stats)
- Specific ask (playlist placement targets: "Buen Vibe", "Tulum Vibes", "Novedades Viernes")
- Assets (links to press photos, audio, etc.)

The task tracker shows "Pitched to DSPs", "Reptich a todas las plataformas", "Repitch DSPs" as recurring tasks for almost every artist.

### The Solution

A **Pitch Drafter** that given an artist + release generates per-platform email drafts with:
- Pre-filled release metadata from the Google Sheets schedule
- Platform-specific playlist targets from the playlist database
- Recent traction data auto-pulled (radio plays, existing playlist placements, press hits)
- Customizable template per platform

### Implementation Path

1. **New module**: `pitch-drafter/draft_pitches.py`
2. **Release data**: Pull from existing Google Sheets schedule (artist, title, date, Spotify URI, ISRC, label)
3. **Platform templates**: One template per DSP, each referencing that platform's relevant playlists from `playlist_database.csv`
4. **Traction section**: Optionally run a quick Soundcharts lookup + press search to include "already receiving radio play on X stations" or "covered by Indie Rocks, Tecoapple" as social proof
5. **LLM assist** (optional): Use Claude to generate a compelling pitch paragraph from the raw data
6. **Web UI**: Select artist + release from dropdown (populated from schedule) → generates draft emails per platform → copy buttons for each
7. **Output**: Plain text / HTML emails, one per platform, ready to paste

### Data Sources
- Release schedule (Google Sheets — already integrated)
- Playlist database (existing — has platform-specific playlists with names and IDs)
- Soundcharts (optional — for traction stats)
- Press results (optional — for social proof)

### Technical Notes
- v1: Pure template fill (no LLM). Fast and predictable.
- v2: Add LLM-generated pitch paragraph for the "why it matters" section.
- The pitch timing matters: proposals mention "two weeks before, one week before, and 1-2 weeks after release." Could tie into the release calendar for automated reminders.

## Opportunity 5: Release Calendar Dashboard ✅ DONE

### Priority: MEDIUM
### Time saved: Reduces coordination overhead across entire team
### Status: Implemented — `/calendar` page with phase tracking, artist filter, action buttons

### The Problem

The team manages 50+ artists with staggered release timelines. The release schedule lives in Google Sheets, but there's no visual timeline showing:
- What's releasing this week / next week / this month
- Which phase each release is in (pre-release pitch, release week, post-release repitch)
- What actions are due for each release (pitch DSPs 2 weeks before, service radio 1 week before, press follow-up 1 week after)
- Who on the team is responsible for what

The task tracker is flat — it doesn't connect tasks to release phases or show the pipeline view.

### The Solution

A **Release Calendar** page in the web UI that:
- Pulls the live release schedule from Google Sheets
- Shows a timeline/calendar view with releases plotted by date
- Color-codes by phase (upcoming → release week → post-release)
- Shows automated action reminders ("PNAU — Rollin': DSP pitch due in 3 days")
- Links each release to quick-run the existing tools (run DSP check, run press search)

### Implementation Path

1. **New route**: `/calendar` in `web/app.py`
2. **Data source**: Existing `load_release_schedule()` from Google Sheets
3. **Phase calculation**: Based on release date vs. today:
   - `-14 days`: Pre-pitch phase (DSP emails should go out)
   - `-7 days`: Radio servicing, press outreach
   - `0 days`: Release day
   - `+7 days`: Post-release repitch, press follow-up
   - `+14 days`: Reporting phase (generate client report)
4. **Frontend**: HTML/CSS calendar grid or timeline view (no JS framework needed — vanilla JS like the existing UI)
5. **Action buttons**: "Run DSP Check", "Run Press Search", "Generate Report" links per release that pre-fill the existing tool forms
6. **Team assignment**: If Notion integration is added later, show who's assigned to each artist

### Technical Notes
- The release schedule already has `week_block` grouping. This maps naturally to a weekly calendar view.
- Could add a simple notification system (browser notifications or a "due today" banner) for upcoming actions.
- This is more of a dashboard/coordination tool than a generator — lower direct time savings but high visibility value for the team lead.

## Opportunity 6: Contact & Outlet Discovery Assistant ✅ DONE

### Priority: MEDIUM
### Time saved: ~1-2 hrs per month of manual scouting
### Status: Implemented — `discovery/discover_outlets.py` + "Discovery" tab in web UI (WIP)

### The Problem

The highlights email shows team members spending significant time discovering new press outlets, content creators, venues, and media contacts across LATAM. Each discovery is manually shared via email and then added to Notion. The process is:
1. Someone stumbles on a new outlet while doing press searches
2. They share it in an email/Slack with a description
3. Someone (usually Monen) adds it to Notion with tags and metadata

The existing Press Pickup tool already identifies outlets "not in DB" — but it only flags them passively during a search. There's no proactive discovery.

### The Solution

A **Discovery Assistant** that:
- Periodically searches for new LATAM music/entertainment outlets not in the current database
- Given a genre or scene (e.g., "electronic music Brazil"), finds blogs, magazines, podcasts, TikTok/IG creators covering that space
- Cross-references against the existing press DB to highlight only genuinely new discoveries
- Generates a formatted "new contacts" digest with outlet name, URL, focus, country, estimated reach

### Implementation Path

1. **New module**: `discovery/discover_outlets.py`
2. **Search strategy**: Use the same SearXNG/Serper/Google News sources already integrated, but with discovery-focused queries:
   - `"blog" OR "magazine" OR "podcast" "música electrónica" site:.mx`
   - `"entrevista" OR "reseña" "indie rock" site:.com.br`
3. **Deduplication**: Check each found domain against the existing `press_database.csv`
4. **Enrichment**: Use Claude to generate a one-line description + estimated genre focus for each new outlet (similar to existing `generate_description_with_llm()`)
5. **Output**: Formatted digest (email-ready or .csv for Notion import)
6. **Web UI**: "Discover new outlets" button with genre/country filters

### Technical Notes
- This is a research/scouting tool, not a reporting tool. Results need human review before being added to the database.
- Could run monthly on a schedule, producing a "New Outlets Digest" email.
- The press DB currently has no genre tags — adding them would make this tool (and the Proposal Generator) much more effective.

## Opportunity 7: Press Release Generator — DONE ✓

### Priority: MEDIUM-LOW
### Time saved: ~30-45 min per release

### The Problem

For every release, the team services a press release in Spanish and Portuguese to their outlet contacts. These PRs follow a standard format:
- Headline: "[Artist] estrena [Track] con [Collaborator]"
- Intro paragraph: artist context, what the track is about
- Key facts: release date, label, available platforms
- Quotes (if provided by the label)
- Links: streaming, social media, press photos
- Boilerplate: artist bio

### The Solution

A **PR Generator** that takes release info (artist, track, date, genre, collaborator, one-sentence description) and generates press releases in both Spanish and Portuguese, ready to send.

### Implementation Path

1. **New module**: `pr-generator/generate_pr.py`
2. **Template**: Fixed structure with variable fills
3. **LLM generation**: Use Claude to write the intro paragraph and translate to Portuguese, given the raw facts
4. **Output**: HTML email body + plain text, in both ES and PT-BR
5. **Web UI**: Form with release details → preview in both languages → copy buttons

### Technical Notes
- This is a good candidate for LLM generation since the output is creative text, not structured data.
- v1 could be template-only (mad-libs style). v2 adds LLM for natural-sounding prose.
- The existing press tool's outlet database could be used to generate a distribution list alongside the PR.

## Opportunity 8: Budget Calculator

### Priority: LOW
### Time saved: ~15-20 min per proposal

### The Problem

Every proposal includes a budget section with line items for radio stations, influencer campaigns, DJ/club servicing, etc. The pricing is relatively fixed (e.g., "Los 40 MX: $7,734 USD", "Energia 97 BR: $2,900 USD"). Someone manually looks up prices and sums them.

### The Solution

An interactive **Budget Calculator** in the web UI where the user selects services from checkboxes:
- Radio: pick stations by country → auto-sum
- Influencers: select tier (macro/micro) and budget range
- Club/DJ servicing: select countries
- DMM base fee: $2K/month × campaign duration

Total updates in real time.

### Implementation Path

1. **Data file**: `data/pricing.json` with all service pricing
2. **Web UI**: Interactive form with country/service checkboxes, real-time total
3. **Export**: "Copy budget section" for pasting into proposals, or embed directly into the Proposal Generator

### Technical Notes
- Purely frontend — no backend needed beyond serving the pricing data.
- Prices need manual updates when they change (probably 1-2x per year).
- Could be a section within the Proposal Generator rather than a standalone tool.

## Architecture Notes

All new tools should follow the existing pattern:
- **Backend**: Python module in its own directory (e.g., `report-compiler/`), with a `run_*()` function that returns structured data
- **Web API**: New endpoints in `web/app.py` using the existing job queue (UUID-based, threaded, polled from frontend)
- **Frontend**: New tab or sub-panel in `web/templates/index.html`, matching existing DMM brand design
- **Output**: .docx via `python-docx`, with optional .txt and .json exports
- **LLM**: Groq Llama 3.3 70B (free, no billing) for AI text generation — used in Discovery (outlet descriptions + type classification), Proposal Generator (Goal/Strategy + Digital Marketing sections), and Weekly Digest (campaign analysis). Gated behind `GROQ_API_KEY` with graceful fallback to templates. Google Gemini used for PR Translator AI mode.

The existing `shared/` module should be extended:
- `shared/database.py` — add loaders for new data files (radio targets, pricing)
- `shared/report_builder.py` — shared .docx generation utilities (branding, section headers, proof image embedding) used by all tools that produce Word documents
